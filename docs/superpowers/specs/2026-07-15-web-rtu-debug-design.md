# ModbusMate 网页调试与 Modbus-RTU 支持设计

日期：2026-07-15  
状态：已完成方案确认，待用户审核设计文档

## 1. 背景与目标

ModbusMate 当前是 Electron 桌面应用，仅支持 Modbus-TCP。为了提高真实设备现场调试效率，需要先提供浏览器调试入口，并增加通过 USB-RS485 连接 Modbus-RTU 设备的能力。

本次目标：

- 保留现有 Modbus-TCP 全部能力。
- 新增跨 macOS 和 Windows 的 Modbus-RTU 支持。
- 调试工作台、设备总览、设备调试、通信日志、类型和实例管理全部支持 TCP 与 RTU。
- 通过 `npm run web` 启动本地服务和浏览器界面，不要求先构建 Electron 安装包。
- Electron 后续复用相同通信核心，不维护两套设备通信逻辑。
- 提供足够完整的实时及落盘日志，让人工反馈可以定位到具体设备、功能码和通信阶段。
- 以《一体24K EMS通讯协议-260409》作为首个真实设备验证样例，但不将该协议参数写死为产品规则。

第一阶段只要求一个 RTU 从站独占一个串口。数据结构需要允许未来扩展为同一 RS485 总线上多个从站共享串口。

## 2. 非目标

- 本阶段不实现云端部署或远程访问本地串口。
- 本阶段不实现多主站访问同一 RTU 总线。
- 本阶段不自动执行任何真实设备写入测试。
- 本阶段不把 EMS 协议的寄存器、波特率或读取限制作为全局默认规则。
- 本阶段不重写现有前端框架。

## 3. 总体架构

```text
浏览器界面
   │ HTTP + WebSocket
本地 Node 调试服务
   ├── 配置与点表服务
   ├── 结构化日志服务
   ├── 轮询与设备管理
   └── Modbus 统一传输接口
          ├── TCP Transport → IP 网络设备
          └── RTU Transport → USB-RS485 → 从站
```

浏览器不能直接稳定访问 Modbus-TCP，也不能跨浏览器一致地管理串口，因此所有真实设备通信由本地 Node 服务完成。浏览器仅负责配置、展示、写入确认和日志查看。

### 3.1 统一传输接口

定义统一的 Modbus 传输能力：

- `connect(config)`
- `disconnect()`
- `reconnect()`
- `read(area, address, count)`
- `write(area, address, words)`
- `connected`
- 连接状态与诊断事件

现有 TCP 逻辑迁移为 `TcpTransport`。新增 `RtuTransport`，继续使用 `modbus-serial` 的 RTU 串口能力。轮询器和设备管理器只依赖统一接口，不根据通信方式编写分支读写逻辑。

### 3.2 本地网页服务

本地服务负责：

- 静态托管现有 `renderer` 页面。
- 提供连接、断开、轮询、写入、设备实例启停等 HTTP API。
- 通过 WebSocket 推送数据、状态和日志事件。
- 枚举本机串口。
- 保存配置、设备类型和设备实例。
- 导入和导出点表。
- 保存结构化运行日志。

新增浏览器适配层 `web-api.js`，向前端暴露与当前 `window.api` 尽量一致的方法，使现有页面可以同时运行于 Electron 和浏览器环境。

### 3.3 Electron 兼容

Electron 继续保留安全的 preload 白名单。通信核心、设备管理、配置校验和日志格式放在与运行外壳无关的模块中。Electron 可以直接调用核心模块，也可以在后续版本内嵌本地服务；本阶段优先完成网页调试链路，不要求立即改变 Electron 启动方式。

## 4. 通信配置模型

设备实例增加 `transport` 字段：

```js
// TCP 实例
{
  transport: 'tcp',
  host: '192.168.137.101',
  port: 8080,
  unitId: 1,
  timeout: 2000
}

// RTU 实例
{
  transport: 'rtu',
  serialPath: 'COM3',
  baudRate: 9600,
  dataBits: 8,
  stopBits: 1,
  parity: 'none',
  unitId: 1,
  timeout: 2000
}
```

TCP 和 RTU 均允许设置轮询周期、超时和从站 ID。RTU 参数必须支持常用波特率、7/8 数据位、1/2 停止位以及 none/even/odd 校验。

串口枚举返回稳定字段：

- `path`
- `manufacturer`
- `serialNumber`
- `vendorId`
- `productId`
- 可供用户识别的显示名称

Windows 串口通常显示为 `COM3`，macOS 通常显示为 `/dev/tty.usbserial-*` 或 `/dev/tty.usbmodem-*`。界面提供刷新按钮，不缓存已经拔出的端口。

## 5. 点表与设备模板

设备类型只保存点表和协议级读取策略；连接参数保存在设备实例。相同点表可以被 TCP 或 RTU 实例复用。

点位模型在现有字段基础上增加：

```js
{
  name: '总有功功率',
  area: 'holding',
  addr: 12523,
  type: 'int16',
  wordOrder: 'AB',
  k: 1,
  b: 0,
  decimals: 0,
  unit: 'W',
  access: 'R',
  category: 'EMS/电表',
  description: '',
  dangerous: false,
  words: 1,
  enumMap: null,
  bitDefinitions: null
}
```

新增数据能力：

- ASCII 字符串，用于厂家、型号和序列号。
- 有符号 16 位数据。
- 故障位字段及枚举值说明。
- 点位读写权限。
- 危险写入标记。
- 点位分类和备注。

`words` 明确点位占用的 16 位寄存器数量，ASCII 字符串按该长度读取；`enumMap` 保存状态值说明；`bitDefinitions` 保存故障字每一位的名称和备注。普通数值类型未填写 `words` 时继续根据数据类型自动推导。

设备类型通过 `maxReadRegisters` 和 `minPollInterval` 配置最大单次读取寄存器数量和最小轮询周期。未配置时使用通用 Modbus 安全默认值；EMS 示例模板配置单次最大 47 个寄存器、建议轮询周期 1000ms。

地址默认按照协议地址直接发送。PLC 显示地址仅是界面显示选项，不能改变底层请求地址。

## 6. EMS 示例模板

首个真实设备样例采用以下配置：

- Modbus-RTU、RS485。
- 默认 9600bps、8 数据位、无校验、1 停止位。
- 从站 ID 1。
- 地址为报文中的直接地址，不做 40001 偏移。
- 支持功能码 02、03、04、05、06、10。
- 协议声明帧长度最大 100 字节，因此读取块最大设置为 47 个寄存器。

模板覆盖设备基础信息、电池基础信息、1 至 8 号电池包、EMS 控制和三相电表点位。恢复出厂、BMS 充电使能、DCDC 开关机等控制点标记为危险写入。

该模板是可删除、可编辑、可导入导出的示例数据，不影响其他 Modbus 设备使用不同参数。

## 7. 网页交互

### 7.1 调试工作台

连接区域增加 TCP/RTU 选择：

- TCP 显示 IP、端口、从站 ID和超时。
- RTU 显示串口、刷新按钮、波特率、数据位、停止位、校验、从站 ID 和超时。

连接成功后沿用现有区域、地址、数量、数据类型、字序、换算公式和写入操作。

### 7.2 设备实例

新增或编辑设备实例时选择通信方式。实例列表和设备总览显示通信摘要，例如：

- `TCP · 192.168.137.101:8080 · ID 1`
- `RTU · COM3 · 9600 8N1 · ID 1`

设备状态继续使用 connected、offline、disconnected 和 error。RTU 断线时区分串口被拔出、端口占用和从站无响应。

### 7.3 写入保护

- `access: R` 的点位不显示写入按钮，后端也拒绝写入。
- 普通写入确认框显示设备、区域、地址、原值和新值。
- 可选写入后立即读回校验。
- `dangerous: true` 使用明显警告，并要求再次确认。
- 系统不自动选择或写入真实设备测试地址。

## 8. 日志设计

本地服务生成结构化日志，并通过 WebSocket 实时推送到网页，同时按日期写入本地文件。

每条通信日志包含：

- 时间和日志级别。
- 设备实例 ID 和名称。
- TCP 或 RTU。
- 连接目标或串口参数。
- 从站 ID。
- 功能码、区域、地址和数量。
- 请求耗时。
- 成功结果、异常码或错误类别。
- 调试模式下的 TX/RX 十六进制报文。

网页日志支持按设备、级别和通信方式筛选并导出。日志文件采用滚动策略，避免长期调试无限占用磁盘。

错误信息需要转换为明确中文，包括：

- 串口不存在、被占用、权限或驱动异常。
- CRC 错误、响应超时、从站 ID 不匹配。
- Modbus 异常码 01、02、03、04 等。
- TCP 连接拒绝、网络不可达和连接中断。
- 点表格式、地址和参数校验错误。

## 9. 数据流

读取流程：

1. 浏览器提交连接或设备启动配置。
2. 本地服务校验配置并创建对应 Transport。
3. 轮询器根据点表生成读取块。
4. Transport 串行执行 Modbus 请求。
5. 服务记录结构化日志并推送原始数据。
6. 浏览器使用 Codec 解码、换算和显示。

写入流程：

1. 浏览器根据点位权限和危险标记展示确认框。
2. 后端再次校验点位权限、区域和值范围。
3. 写任务进入设备串行队列，优先于下一次轮询。
4. 写入完成后按配置立即读回。
5. 日志记录写入和校验结果。

## 10. 测试策略

### 10.1 自动测试

- TCP/RTU 配置校验。
- Transport 统一接口行为。
- RTU 读写、超时、重连、端口占用和拔出场景。
- 通过依赖注入或虚拟串口测试，不要求 CI 接入真实设备。
- 点表读取块拆分和设备自定义上限。
- ASCII、Int16、故障位、枚举和倍率解析。
- 只读与危险点位的前后端保护。
- Web HTTP API 和 WebSocket 事件。
- 配置迁移：旧 TCP 实例缺少 `transport` 时按 TCP 处理。
- 现有 77 项测试全部回归。

### 10.2 真实设备验收

1. macOS 枚举 USB-RS485 串口。
2. 使用 9600/8N1、从站 ID 1 连接 EMS。
3. 首先只读基础信息和 EMS 电表数据。
4. 核对地址、倍率、符号、字符串和故障位。
5. 根据日志确认请求、响应、耗时和异常信息。
6. 只有用户明确确认安全地址后才执行写入测试。
7. 验证拔插串口、超时、停止轮询和重新连接。
8. 在 Windows 验证 COM 口枚举、连接和构建。

## 11. 实施阶段

### 阶段一：通信核心

- 抽象统一 Transport。
- 迁移现有 TCP 实现。
- 新增 RTU、串口枚举及配置校验。
- 调整轮询器和设备管理器。

### 阶段二：网页调试服务

- 增加 HTTP/WebSocket 本地服务。
- 增加浏览器 API 适配层。
- 让现有全部页面支持网页模式和 RTU 配置。
- 增加结构化落盘日志。

### 阶段三：模板与真实设备验收

- 生成可导入的 EMS 示例点表。
- 完成 macOS 真实设备只读验证。
- 经用户确认后验证安全写入。
- 完成 Windows 串口与构建验证。

## 12. 完成标准

- `npm run web` 可以启动本地调试服务和网页。
- macOS 与 Windows 可以枚举 USB 串口。
- 所有现有页面均能配置和使用 TCP 或 RTU 设备。
- TCP 原有功能和测试不退化。
- RTU 可以稳定执行读、写、轮询、停止和重连。
- 点表可以表达权限、危险写入、ASCII、Int16 和故障位。
- 普通日志实时显示并持久化，通信失败能够定位到具体阶段。
- EMS 作为示例模板完成真实设备只读验收。
